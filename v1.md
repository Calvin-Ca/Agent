# AI 模型全生命周期管理平台 — 需求规格说明书 v1

## 1. 项目定位

面向 AI 落地团队的一站式平台，覆盖 **数据 → 训练 → 评估 → 部署** 全流程，解决以下痛点：

- 数据版本混乱、标注质量不可追溯
- 训练实验无法复现（代码、数据、超参三者版本未对齐）
- 模型评估缺乏标准化流程，指标散落在 notebook 中
- 部署环节与训练脱节，模型交付依赖口头沟通

**一个 Project = 一个 AI 落地项目**，包含该项目从数据准备到模型上线的全部资产。

## 3. 数据集管理（Dataset）

管理训练/验证/测试数据的全生命周期。

### 3.1 数据集元信息

| 字段 | 说明 |
|------|------|
| 名称 / 描述 | 数据集基本信息 |
| 任务类型 | 分类、检测、分割、NLP、多模态等 |
| 数据格式 | COCO、VOC、YOLO、CSV、自定义 |
| 存储路径 | 本地路径 / MinIO / S3 |
| 标签体系 | 类别列表、层级关系、颜色映射 |

### 3.2 类别管理

- 类别定义：名称、ID 映射、父子层级（支持多级分类）
- 类别映射：不同版本间的类别对齐（合并/拆分/重命名）
- 类别统计：各类别的样本数量、占比、分布直方图

### 3.3 版本管理

- 版本号：语义化版本（v1.0.0）
- 变更记录：每个版本的变更内容（新增样本数、删除数、标注修改数）
- 版本对比：任意两版本间的 diff（新增/删除/修改的样本列表）
- 数据快照：每版本对应的文件清单哈希，确保可复现
- 拆分策略：记录 train/val/test 的拆分比例和随机种子

### 3.4 可视化

- 样本浏览：图片缩略图 + 标注叠加显示（bbox / mask / keypoint）
- 分布统计：类别分布、图片尺寸分布、标注密度热力图
- 数据质量：标注异常检测（空标注、超出边界、面积过小）
- 对比视图：版本间 / 拆分间的分布差异对比

### 3.5 数据处理

- 格式转换：COCO ↔ VOC ↔ YOLO 互转
- 数据增强预览：对选中样本实时预览增强效果
- 数据清洗：去重（perceptual hash）、异常标注过滤

## 4. 模型管理（Model）

管理模型架构、权重文件、配置的版本化资产。

### 4.1 模型注册

| 字段 | 说明 |
|------|------|
| 模型名称 / 别名 | 便于检索 |
| 任务类型 | 与数据集任务类型对应 |
| 框架 | PyTorch / TensorFlow / ONNX / TensorRT |
| 来源 | 自研 / HuggingFace / ModelScope / 预训练 |
| 基座模型 | 基于哪个预训练模型微调（溯源链） |

### 4.2 架构可视化

- 网络结构图：自动解析模型定义，渲染计算图（支持折叠/展开子模块）
- 参数统计：总参数量、可训练参数量、各层参数量占比
- FLOPs 估算：给定输入尺寸，估算推理计算量
- 结构对比：两个模型架构的 diff 视图

### 4.3 权重管理

- 权重版本：每次训练产出的 checkpoint，关联训练实验 ID
- 权重格式：`.pt` / `.safetensors` / `.onnx` / `.trt`
- 存储策略：本地 + 对象存储（MinIO/S3），支持自动清理旧版本
- 权重继承：记录 from_pretrained 链路（预训练 → 微调 → 蒸馏）
- Best 标记：自动或手动标记最优权重（依据验证指标）

### 4.4 超参数管理

- 超参数模板：每种任务类型预设推荐参数模板
- 参数版本化：每次实验的完整超参快照（YAML/JSON）
- 参数搜索空间：定义可搜索范围（为训练管理中的 HPO 做准备）
- 参数对比：多次实验的参数差异表格

### 4.5 预处理管理

- 预处理流水线：有序的处理步骤列表（resize → normalize → augment → ...）
- 每步配置：参数化配置（如 resize 目标尺寸、normalize 均值方差）
- 版本绑定：预处理配置与模型版本绑定，确保训练/推理一致
- 预览功能：对选中样本实时预览预处理效果

## 5. 训练管理（Training）

管理训练实验的全过程，确保可复现。

### 5.1 实验（Experiment）

一个实验 = 一次完整的训练运行，记录以下信息：

| 维度 | 记录内容 |
|------|---------|
| 数据 | 数据集版本 + 拆分策略 + 预处理配置 |
| 模型 | 模型名称 + 初始权重版本 |
| 超参 | 完整超参快照 |
| 环境 | GPU 型号、CUDA 版本、Python 包版本（pip freeze） |
| 代码 | Git commit hash（可选关联） |

### 5.2 训练执行

- 启动方式：Web UI 提交 / CLI 提交 / API 调用
- 资源管理：GPU 选择、显存限制、多卡策略（DP / DDP / FSDP）
- 任务队列：支持排队（GPU 占用时自动等待）
- 断点续训：从指定 checkpoint 恢复
- 提前终止：手动停止 / Early Stopping（patience + metric）

### 5.3 训练监控

- 实时指标：loss、lr、metric 曲线（逐 step / 逐 epoch）
- 资源监控：GPU 利用率、显存、CPU、内存
- 日志流：实时 stdout/stderr 输出
- 通知：训练完成 / 异常终止时推送通知

### 5.4 实验对比

- 多实验曲线叠加：在同一坐标系下对比 loss / metric
- 参数差异表：自动高亮不同实验间的参数差异
- 排行榜：按指定指标排序，展示 Top-N 实验

### 5.5 微调（Fine-tuning）

- 冻结策略：支持配置冻结层（freeze backbone / freeze N layers）
- LoRA / QLoRA：低秩适配器微调，记录 adapter 权重独立版本
- 学习率策略：分层学习率（backbone 小 lr，head 大 lr）

## 6. 测试管理（Test / Evaluation）

标准化模型评估流程，确保评估结果可复现、可对比。

### 6.1 评估任务

| 字段 | 说明 |
|------|------|
| 关联模型 | 模型名称 + 权重版本 |
| 关联数据集 | 数据集名称 + 版本 + 拆分（test split） |
| 评估指标 | 任务相关指标集合（见 6.3） |
| 预处理配置 | 必须与训练时一致（从模型版本中继承） |

### 6.2 环境管理

- 环境快照：评估时的硬件信息（GPU 型号、驱动版本）+ 软件环境（Python 包版本）
- 环境复现：支持导出/导入环境配置（requirements.txt / conda env）
- 推理后端：PyTorch / ONNX Runtime / TensorRT / OpenVINO

### 6.3 评估指标

按任务类型内置常用指标：

| 任务类型 | 指标 |
|---------|------|
| 分类 | Accuracy、Precision、Recall、F1、AUC-ROC、Confusion Matrix |
| 检测 | mAP@50、mAP@50:95、AP per class、Precision-Recall 曲线 |
| 分割 | mIoU、Dice、Pixel Accuracy、per-class IoU |
| NLP | BLEU、ROUGE、Perplexity、Exact Match |
| 通用 | 推理速度（FPS / Latency）、显存占用、模型大小 |

支持自定义指标（用户提供计算函数）。

### 6.4 结果可视化

- 指标仪表板：雷达图总览、柱状图对比
- 预测可视化：图片 + 预测叠加（bbox / mask / heatmap），支持 GT vs Pred 对比
- 错误分析：按置信度 / 类别 / 图片尺寸维度分析错误样本
- 混淆矩阵：交互式热力图，点击查看具体样本
- 阈值分析：Precision-Recall 随阈值变化曲线

### 6.5 版本管理

- 评估记录版本化：每次评估运行生成唯一 ID
- 结果对比：任意多次评估结果的指标对比表
- 回归检测：与基线版本对比，自动标记指标退化（红）/ 提升（绿）
- 评估报告：一键生成评估报告（Markdown / PDF），包含指标、可视化、环境信息

## 7. 部署管理（Deploy）

管理模型从实验到生产的交付过程。

### 7.1 模型优化

| 优化手段 | 说明 |
|---------|------|
| 量化 | FP16 / INT8 / INT4，支持 PTQ（训练后量化）和 QAT（量化感知训练） |
| 剪枝 | 结构化剪枝（通道/层）、非结构化剪枝（权重） |
| 蒸馏 | Teacher → Student 蒸馏，记录 Teacher 模型版本 |
| 格式转换 | PyTorch → ONNX → TensorRT / OpenVINO / CoreML |

每次优化自动运行评估，对比优化前后指标 + 性能变化。

### 7.2 部署包管理

- 部署包 = 模型权重 + 预处理配置 + 后处理配置 + 推理代码 + 环境依赖
- 版本化：每个部署包有唯一版本号，关联源模型权重版本
- 构建产物：Docker 镜像 / 压缩包 / 模型仓库格式（Triton Model Repository）

### 7.3 部署目标

| 目标 | 方式 |
|------|------|
| 服务化 | REST API / gRPC，支持 Triton Inference Server |
| 边缘端 | ONNX Runtime / TFLite / TensorRT，导出适配格式 |
| 批量推理 | Celery 任务队列，大规模离线推理 |

### 7.4 服务监控

- 性能指标：QPS、P50/P95/P99 延迟、GPU 利用率
- 模型漂移：定期对比线上预测分布与训练集分布
- A/B 测试：多版本并行服务，按流量比例分配
- 回滚：一键回滚到上一个稳定版本

## 8. 全局功能

### 8.1 血缘追踪（Lineage）

任意模型权重可追溯完整链路：

```
数据集 v1.2 + 预处理 v3 + 超参 config_a.yaml
       ↓
  训练实验 EXP-0042（GPU: A100, 50 epochs）
       ↓
  权重 best_model_ep38.pt（val mAP=0.847）
       ↓
  评估 EVAL-0015（test mAP=0.831, FPS=45）
       ↓
  量化 INT8 → deploy_v2.1.trt（test mAP=0.825, FPS=120）
       ↓
  部署 DEPLOY-0008（Triton, 线上服务中）
```

### 8.2 权限管理

| 角色 | 权限 |
|------|------|
| Admin | 全部操作 + 用户管理 |
| ML Engineer | 训练 + 评估 + 部署 |
| Data Engineer | 数据集管理 + 标注 |
| Viewer | 只读（查看实验/指标/报告） |

### 8.3 通知与协作

- 事件通知：训练完成、评估异常、部署上线（WebSocket / 邮件 / Webhook）
- 实验批注：对实验、评估结果添加评论和标签
- 审批流：模型上线前的人工审批节点

## 11. 与现有项目的关系

本需求中的"Project"概念与当前 Smart Weekly Report Agent 项目中的"Project"定义不同：

| | 当前项目 | 本需求 |
|---|---|---|
| Project 含义 | 工程项目（施工/建设） | AI 落地项目（模型训练部署） |
| 核心实体 | Progress / Report / Document | Dataset / Model / Experiment / Evaluation / Deployment |
| Agent 角色 | 生成周报、回答进度问题 | 辅助训练调参、自动评估、智能部署建议（后续扩展） |

建议作为 **独立项目** 开发，可复用现有项目的：
- 九层架构设计模式
- 工具注册制（`BaseTool` + `ToolRegistry`）
- 基础设施层（认证、中间件、异常处理、配置管理）
- Celery 异步任务框架
- MinIO 对象存储集成

### 12.1 目录设计说明

- `api/` 保持“薄接口”原则，只负责参数校验、权限控制和路由装配，不承载复杂业务逻辑。
- `agents/` 负责 Agent 状态机、节点编排和 Prompt 管理，是系统的核心认知与决策层。
- `tools/` 负责所有可执行能力的注册、校验与运行，强调标准化接口和可扩展性。
- `model_service/` 负责适配不同模型后端，避免业务层直接耦合 OpenAI、vLLM、Ollama 等实现。
- `memory/` 将短期会话记忆、长期向量记忆、摘要压缩统一起来，便于后续演进为可配置记忆策略。
- `services/` 承接业务编排逻辑，作为 API 与 Agent/Tool/DB 之间的稳定中间层。
- `observability/` 独立承载日志、链路追踪、指标和成本统计，便于后期接入企业级监控系统。

### 12.2 可扩展能力预留

以下能力可作为后续扩展方向，建议以插件化或独立 workflow 的方式接入：

- `yolo`：目标检测训练 / 推理工具链
- `sort`：多目标跟踪能力
- `resnet3d`：视频时序建模或动作识别
- `progress_recognition`：进度识别 Agent / pipeline
- `file_chat`：面向文档、图纸、报告的文件问答能力
